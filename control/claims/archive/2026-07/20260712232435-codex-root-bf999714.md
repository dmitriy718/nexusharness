# Implement structured run failure diagnostics and correction UX

## Summary

Implemented typed runtime failure classification, persisted run-failure evidence, legacy failure interpretation, and prominent correction guidance across run detail and history.

## Files changed

server/runtimeAdapters.ts, server/agentLoop.ts, server/types.ts, src/api/types.ts, src/features/runs/RunDetailPage.tsx, RunsPage.tsx, runModel.ts, tests/runtimeFailure.test.ts, tests/runModel.test.ts, CHANGELOG.md, plus generated control evidence.

## Verification

Root cause reproduced from persisted timing/configuration: healthy Ollama, 60000 ms deadline, three parallel Executors, 45-second Planner. Seventeen focused tests passed for timeout/HTTP/cancellation classification, structured corrections, legacy interpretation, and UI. Both TypeScript builds, lint, live desktop/mobile inspection, and two full release:verify passes passed; final total 278 tests plus all browser/smoke/package gates.

## What worked

Future failures retain typed phase, agent, subtask, runtime, model, endpoint, deadline, retryability, technical detail, and corrections. Legacy aborted errors render as timeouts immediately. Runtime response bodies remain excluded. Run history shows the failure title and detail pages show what happened, where, corrections, direct settings links, and expandable technical evidence.

## What did not work

An initial component test lacked jsdom and an overloaded HTTP fixture type needed an explicit RequestListener; both were corrected. The first targeted axe call used a Page without an explicit BrowserContext and was rerun correctly.

## Unfinished work

The operator should raise the Ollama timeout and reduce parallel Executors before retrying the saved run. Restart NexusHarness to activate backend structured persistence; the rebuilt frontend already improves the existing legacy record.
