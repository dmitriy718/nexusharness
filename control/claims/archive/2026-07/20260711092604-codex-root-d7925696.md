# Implement bounded run history and complete older-run detail contracts

## Summary

Implemented bounded searchable run history with complete older-run detail context, deep links, and duplication.

## Files changed

server/runHistory.ts; server/index.ts; src/api/types.ts; src/features/runs/RunsPage.tsx; src/features/runs/RunDetailPage.tsx; src/styles/pages.css; src/styles/responsive.css; tests/runHistory.test.ts; tests/performance.browser.test.ts; tests/productionHarness.ts; CHANGELOG.md; control/issues/items/NH-0007.md; control/issues/BOARD.md

## Verification

Perspective 1 contract/model: 80 tests validate query defaults, 100-row maximum, invalid offsets/status, filter-before-page behavior, totals, and hasMore; ESLint and both TypeScript builds pass. Perspective 2 production browser: 1,500-run history ready in 88 ms with 100 rows, expanded to 200, opened a record outside compact state with full detail, and duplicated its task successfully. Full release verification also passed five accessibility, six visual, and four performance suites; npm audit found 0 production vulnerabilities.

## What worked

Dedicated /api/runs paging avoids loading the entire store, and /api/runs/:id supplies the run plus linked audit and approval context. RunsPage debounces server filtering and loads bounded pages. RunDetail renders compact state immediately when available, then refreshes from the complete endpoint; older duplicates fetch their source record explicitly.

## What did not work

Two earlier narrow claims were intentionally released without edits after dependency audits showed that UI-only paging and list-only API paging would leave older rows unavailable or broken on detail navigation.

## Unfinished work

None. Large audit and run histories are both bounded and measured; NH-0007 can return to verification.
