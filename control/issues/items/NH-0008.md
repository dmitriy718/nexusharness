# Meet the v2 WCAG 2.2 AA accessibility gate

## User impact

Unlabeled fields, absent focus rules, unannounced state, and non-semantic tree behavior block keyboard and assistive-technology users.

## Acceptance criteria

- All controls have persistent programmatic labels and errors.
- Keyboard, focus, live-region, dialog, drawer, and tree behavior is documented and tested.
- Contrast, touch targets, reduced motion, reflow, and zoom meet the v2 plan.
- Automated checks and an NVDA walkthrough pass.

## Dependencies and blockers

All UI phases; this issue remains a release gate through final verification. HR-002, the owner’s NVDA walkthrough, is now the remaining acceptance check.

## Progress

The automated candidate closes the observed programmatic-label, focus, keyboard, live-region, status, contrast-token, touch-target, reduced-motion, reflow, zoom, dialog, and workspace-disclosure gaps. The release gate now runs axe against 18 production routes plus Focus, Studio, Orchestrate, runtime setup, MCP setup, memory editor, and audit-dialog states. Separate browser checks cover route focus, focus trap/return, arrow-key tabs and workspace navigation, 320 CSS px, 200% content zoom, a 400% device-scale equivalent, forced-colors focus and status preservation, coarse-pointer targets, and reduced motion. Semantic colors are also checked at 4.5:1 or better against the lightest application surface.

Automated evidence: 84 unit/component/model tests pass, six production accessibility suites pass, the production build passes, and the route scans report zero detectable WCAG A/AA violations. Axe reports complex-background contrast nodes as manual-review candidates; the semantic-token test and HR-002 cover that limitation rather than treating incomplete automation as a pass. HR-002 remains the only uncompleted acceptance check for this issue.
