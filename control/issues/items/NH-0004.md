# Renovate approval safety and audit observability

## User impact

Raw JSON and visually equivalent decisions make high-risk operations hard to assess and easier to approve accidentally.

## Acceptance criteria

- Approvals show run, agent, risk, targets, command or diff, and consequence.
- Decision controls are separated, in-flight protected, conflict safe, and durable.
- Pending approvals are globally visible.
- Audit events are filterable and show time, actor, action, risk, status, run, and target.
- Sensitive values are redacted from formatted and exported views.

## Dependencies and blockers

NH-0001 and shared diff/command components.

## Progress

Approval records now retain their originating run and subtask. Shell approvals expose the command, resolved working directory, and shell; write approvals include a pre-execution unified diff, byte counts, and before/after hashes; delete approvals identify the target type and clearly warn about recursive deletion. The review surface links back to its run, separates formatted review from recursively redacted raw data, and targets rejection at the originating waiting run without depending on error-message parsing. The dedicated audit route now loads the full 5,000-event local ledger, filters and sorts across actor/risk/status/action and free text, resolves run and target context, windows large results, and provides formatted/redacted detail plus a clearly scoped redacted JSON export.

## Verification evidence

- Contract/model perspective: lint, 36 automated tests across three files, and the production build passed. Tests cover target and command extraction, action summaries, diff classification, and nested credential redaction.
- Lifecycle/browser perspective: an isolated real write request left the file unchanged before approval, persisted its run/subtask, diff, and hashes, and rendered the origin link and colored diff without horizontal overflow. The raw view hid a seeded secret, and rejecting the approval failed the directly linked waiting run even when its error did not contain the approval ID.
- Audit model/API perspective: 42 tests cover direct and run-log linkage, target extraction, compound filters, chronological sort, nested/inline/URL credential redaction, explicit export metadata, and relative time. The isolated API returned the complete fixture ledger independently from compact state.
- Audit rendered perspective: production Chrome at 1440 px rendered all seven operator fields, six labeled filters, three run links, and the detail drawer. Long paths remained contained; nested, bearer-token, and URL credentials were absent from both formatted and raw review. At 390 px the controls collapsed to one column, the page remained visually contained, and the wide ledger stayed inside its own horizontal scroller.
