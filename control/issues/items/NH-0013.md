# Build the transactional execution and proof foundation

## User impact

Agent exploration currently relies on workspace-bounded tools and approvals, but it does not yet provide a transaction boundary that can predict effects, prove what occurred, or commit and roll back an isolated result atomically. This foundation must exist before parallel solution search or broader computer control can be enabled safely.

## Acceptance criteria

- Versioned schemas define objectives, contracted actions, capability leases, expected effects, observed effects, receipts, cell specifications, cell lifecycle, and commit receipts.
- Runtime validation rejects malformed contracts, expired leases, undeclared consequential capabilities, unsupported schema versions, and unsafe repository-relative scopes.
- Every contract and receipt has a deterministic canonical digest suitable for later attestation chaining.
- A provider-neutral execution-cell interface separates transaction semantics from Git worktrees, Windows isolation, Linux microVMs, or future remote providers.
- The first portable provider creates and tears down disposable repository worktrees, records the immutable base revision, detects stale-base commit attempts, and never describes worktree separation as a security boundary.
- A capability broker is the only supported entry point for contracted mutating actions and records predicted-versus-observed effects.
- Existing direct tool behavior remains compatible until the broker migration is complete; compatibility mode is explicit and auditable.
- Unit and integration tests cover schema validation, digest stability, lease expiry, path boundaries, lifecycle transitions, teardown, crash recovery, stale-base rejection, and unchanged-primary-workspace behavior.
- Operator UI exposes lifecycle, provider, base revision, capability envelope, effect variance, verification state, and commit/rollback availability without presenting unfinished isolation as hardened.
- Documentation distinguishes process/worktree isolation from Windows Sandbox and Firecracker security boundaries.

## Delivery slices

1. **Contracts:** typed domain model, validators, canonical digests, lifecycle rules, and tests.
2. **Broker:** contracted-action enforcement, short-lived leases, effect receipts, compatibility adapter, and audit linkage.
3. **Portable cells:** disposable locked worktrees, base compare-and-swap, normalized diffs, teardown, and recovery.
4. **Operator surface:** execution inspector, effect comparison, verification evidence, and guarded commit/rollback controls.
5. **Hardened providers:** Windows and Linux adapters plus adversarial isolation review; these remain opt-in until separately verified.

## Dependencies and blockers

- v2.0.0 remains separately gated by HR-002 and HR-003. This v2.1 work must not be used to imply those checks passed.
- The portable worktree provider is a transaction-isolation mechanism, not a host-security sandbox.
- Hardened provider defaults require platform-specific threat modeling and a separate security review.
- Branch-and-prove search, adaptive routing enforcement, and semantic desktop control must not depend on unfinished or advisory-only contracts.

## Verification strategy

- Perspective one: deterministic model and validator tests, including mutation/fuzz cases for contracts and scopes.
- Perspective two: real temporary Git repositories proving primary-workspace immutability, cleanup, commit conflict behavior, and recovery after interrupted cells.
- Perspective three: adversarial broker tests attempting undeclared paths, commands, network destinations, expired leases, and receipt tampering.
- Perspective four: production-browser checks for truthful lifecycle and approval language once the operator surface exists.

## Progress

Delivery slice 1 is complete. The provider-neutral v1 model now covers objectives, normalized capability envelopes, time-bounded leases, contracted actions, expected and observed effects, cell specifications and live records, snapshots, effect sets, action receipts, commit receipts, finite lifecycle transitions, and the provider interface. Contract authorization binds one active lease to the same objective and cell, prevents a contract from outliving or exceeding that lease, and produces normalized SHA-256 digests from plain JSON data.

Validation rejects unsupported versions, unknown fields, expired/used/revoked leases, undeclared risk capabilities, unsafe repository paths, unsafe network origins, non-finite or non-JSON digest input, blocking variance hidden inside a successful receipt, and stale or contradictory commit receipts. The portable provider, operator UI, and hardened adapters remain unimplemented and are not advertised as available.

Live migration review corrected the delivery order. Harness-enforced file reads, writes, and deletes can be redirected into a portable cell, but a Git worktree cannot enforce hostile-process or network denial for model-supplied shell commands, and it cannot roll back MCP remote effects. Full tool migration must therefore not precede hardened execution or claim that `networkDefault: deny` is an operating-system boundary. The safe sequence is run persistence and deterministic file routing, hardened providers, then shell/validation migration and explicit MCP compensation or remote-provider semantics.

The first live integration adapter is complete. `createRunTransactionService` binds every lifecycle publication to one trimmed run identity, while the durable store sink rejects unknown runs, cross-cell rebinding, malformed timestamps, and out-of-order summaries. Attached summaries are cloned before storage, and run timestamps advance without regressing. Four focused persistence tests plus the nine transaction-service tests pass alongside strict lint and both production TypeScript builds. Deterministic file dispatch remains the next sub-slice; no existing shell or MCP behavior is falsely labeled transactional.

Approval-safe transaction admission is complete. Providers can optionally preflight an action inside the isolated cell context before the transaction service enters `executing`. The portable provider validates cell, contract, and lease identity under its cell lock and exposes the owned worktree only to the trusted action-admission adapter. If approval or another prerequisite stops admission, the cell remains `isolated` or `verifying`, the action executor is never called, no receipt is consumed, and the operation can be retried after operator input. Focused verification passes 51 transaction-service, portable-provider, and contract tests plus strict lint and both production builds. The remaining file-action adapter must use this hook rather than requesting approval after execution begins.

The deterministic portable file-write executor is complete. Approval and mutation are split by a digest-bound plan containing the relative path, before/after hashes and sizes, bounded diff, and in-memory content. Contract admission rejects unregistered actions, payload mismatch, authority mismatch, and missing predicted create/update effects before operator approval. Execution re-resolves the cell path and refuses time-of-check/time-of-use drift, then runs through the enforced capability broker and records the actual create/update digest without placing raw content in broker audit metadata. A post-operation state check still reports an applied effect if a later audit write fails, keeping receipts aligned with filesystem truth. Four dedicated adversarial tests plus transaction-service and real portable-provider suites pass (28 focused tests total), along with strict lint and both production builds. Agent-loop dispatch is not connected yet, and no shell or MCP action is represented by this file-only executor.

Windows hardened-provider work has started with a launcher boundary, not a falsely complete provider. The candidate verifies platform and launcher presence, writes an exclusive temporary `.wsb` profile outside the mapped cell, maps exactly one non-root cell folder, disables vGPU, network, audio/video input, printers, and clipboard, enables Protected Client, bounds memory and runtime, launches with a reduced environment and hidden host process window, honors aborts, rejects overlapping configuration paths, and removes the profile on success or failure. Its `securityBoundary` remains `false` until the real-host checkpoint passes. Eleven focused tests cover hardened XML, escaping, unsafe bootstrap names, root/resource rejection, probe truthfulness, runner invocation, cleanup, and mapped/config path separation; lint, builds, and version identity pass.

The Windows host exposes `WindowsSandbox.exe` and DISM reports `Containers-DisposableClientVM` enabled. `npm run test:windows-sandbox` now creates a disposable owner-run probe that checks seed visibility, mapped writeback, Sandbox guest identity, disabled TCP egress, automatic shutdown, host result visibility, and cleanup. HR-004 records the required interactive evidence. This launcher still lacks execution-cell record persistence, brokered command transport, effect harvesting, and atomic promotion; those remain required before it implements `ExecutionCellProvider` or becomes selectable by live runs.

The first HR-004 attempt exposed a real asynchronous-launch defect rather than a missing host feature. `WindowsSandbox.exe` handed the configuration to the Sandbox service and exited successfully before the guest initialized; the launcher treated that short-lived process exit as completion, deleted the `.wsb` profile, and the probe removed its mapped folder. Windows Sandbox then reported `0x80070003` and the host could not find `result.json`. Launch completion is now bound to one validated guest-written completion filename. The launcher removes any stale sentinel before start, retains configuration and mapped data after host process exit, polls abortably within the original total deadline, and only cleans the profile after the sentinel appears or the launch fails. Four new regression/validation cases bring the focused launcher suite to 15 passing tests; lint and both builds pass. HR-004 remains open for a retry after this repair is pushed.

The second HR-004 attempt advanced through guest execution and exposed shutdown ownership. The bootstrap wrote its result and then invoked forced guest shutdown, which the Windows Sandbox client surfaced as connection loss `0x80072746`; host cleanup then encountered `EBUSY` while the mapped cell remained mounted. The probe now snapshots pre-existing `WindowsSandboxRemoteSession` process IDs, never requests guest shutdown, closes only session IDs created by the probe, preserves unrelated user sessions, and retries Windows transient directory locks for a bounded 15 seconds. Both diagnostic captures are retained locally but excluded from npm packages. The 15 focused launcher tests, lint, builds, and package manifest checks pass; HR-004 remains open for a third real-host attempt after push.

The third HR-004 attempt successfully returned all four guest boundary assertions as true under `WDAGUtilityAccount`: seed visibility, mapped write, disabled TCP egress, and Sandbox identity. Host decoding alone failed because Windows PowerShell 5.1 `Set-Content -Encoding UTF8` emitted a byte-order mark that Node's strict `JSON.parse` rejected. The bootstrap now writes JSON with `System.Text.UTF8Encoding(false)`, and the shared parser strips a leading BOM defensively for compatibility with already-produced output. A dedicated plain/BOM regression brings the focused launcher suite to 16 passing tests; lint, builds, and package verification pass. HR-004 remains open for one confirmation run that can also evaluate host writeback and print the final `passed` field.

The fourth HR-004 attempt stopped safely before launching a VM because the preflight `Get-Process` query returned exit code 1 for the normal state in which no `WindowsSandboxRemoteSession` existed. Session discovery now wraps results in an explicit PowerShell array, emits IDs only when present, and terminates with explicit success. Parsing is shared and accepts only positive safe-integer PIDs. A focused empty/multiple/invalid session test brings the launcher suite to 17 passing tests, and the exact child-PowerShell query was separately executed on this host with `queryExit=0`; lint and both builds pass. No Sandbox boundary behavior regressed in this attempt.
