# Build the transactional execution and proof foundation

## User impact

Agent exploration currently relies on workspace-bounded tools and approvals, but it does not yet provide a transaction boundary that can predict effects, prove what occurred, or commit and roll back an isolated result atomically. This foundation must exist before parallel solution search or broader computer control can be enabled safely.

## Acceptance criteria

- Versioned schemas define objectives, contracted actions, capability leases, expected effects, observed effects, receipts, cell specifications, cell lifecycle, and commit receipts.
- Runtime validation rejects malformed contracts, expired leases, undeclared consequential capabilities, unsupported schema versions, and unsafe repository-relative scopes.
- Every contract and receipt has a deterministic canonical digest suitable for later attestation chaining.
- A provider-neutral execution-cell interface separates transaction semantics from Git worktrees, Windows isolation, Linux microVMs, or future remote providers.
- The first portable provider creates and tears down disposable repository worktrees, records the immutable base revision, detects stale-base commit attempts, and never describes worktree separation as a security boundary.
- A capability broker is the only supported entry point for contracted mutating actions and records predicted-versus-observed effects.
- Existing direct tool behavior remains compatible until the broker migration is complete; compatibility mode is explicit and auditable.
- Unit and integration tests cover schema validation, digest stability, lease expiry, path boundaries, lifecycle transitions, teardown, crash recovery, stale-base rejection, and unchanged-primary-workspace behavior.
- Operator UI exposes lifecycle, provider, base revision, capability envelope, effect variance, verification state, and commit/rollback availability without presenting unfinished isolation as hardened.
- Documentation distinguishes process/worktree isolation from Windows Sandbox and Firecracker security boundaries.

## Delivery slices

1. **Contracts:** typed domain model, validators, canonical digests, lifecycle rules, and tests.
2. **Broker:** contracted-action enforcement, short-lived leases, effect receipts, compatibility adapter, and audit linkage.
3. **Portable cells:** disposable locked worktrees, base compare-and-swap, normalized diffs, teardown, and recovery.
4. **Operator surface:** execution inspector, effect comparison, verification evidence, and guarded commit/rollback controls.
5. **Hardened providers:** Windows and Linux adapters plus adversarial isolation review; these remain opt-in until separately verified.

## Dependencies and blockers

- v2.0.0 remains separately gated by HR-002 and HR-003. This v2.1 work must not be used to imply those checks passed.
- The portable worktree provider is a transaction-isolation mechanism, not a host-security sandbox.
- Hardened provider defaults require platform-specific threat modeling and a separate security review.
- Branch-and-prove search, adaptive routing enforcement, and semantic desktop control must not depend on unfinished or advisory-only contracts.

## Verification strategy

- Perspective one: deterministic model and validator tests, including mutation/fuzz cases for contracts and scopes.
- Perspective two: real temporary Git repositories proving primary-workspace immutability, cleanup, commit conflict behavior, and recovery after interrupted cells.
- Perspective three: adversarial broker tests attempting undeclared paths, commands, network destinations, expired leases, and receipt tampering.
- Perspective four: production-browser checks for truthful lifecycle and approval language once the operator surface exists.

## Progress

Delivery slice 1 is complete. The provider-neutral v1 model now covers objectives, normalized capability envelopes, time-bounded leases, contracted actions, expected and observed effects, cell specifications and live records, snapshots, effect sets, action receipts, commit receipts, finite lifecycle transitions, and the provider interface. Contract authorization binds one active lease to the same objective and cell, prevents a contract from outliving or exceeding that lease, and produces normalized SHA-256 digests from plain JSON data.

Validation rejects unsupported versions, unknown fields, expired/used/revoked leases, undeclared risk capabilities, unsafe repository paths, unsafe network origins, non-finite or non-JSON digest input, blocking variance hidden inside a successful receipt, and stale or contradictory commit receipts. The portable provider, operator UI, and hardened adapters remain unimplemented and are not advertised as available.

Evidence: 27 focused contract tests pass. The independent full release gate also passes 111 unit/component/model tests, six accessibility suites, three consequential browser workflows, 36 visual baselines, four performance suites, production smoke, package integrity, build, lint, version identity, and control verification.

Delivery slice 2 is complete. New contracted actions now have one broker entry point with explicit enforced/compatibility modes. The broker validates contract/lease linkage, requires a same-version policy decision, atomically claims single-use leases, obtains effects from an external observer rather than an operation/model response, blocks missing/forbidden/unexpected/mismatched effects, hashes canonical output, chains receipts per cell, and emits audit summaries without payload or raw errors. Policy-evaluator outages fail closed without consuming the lease; observer crashes create failed receipts without leaking error text. Existing local tools remain on their pre-contract path and are not falsely described as broker-enforced.

Evidence: 16 focused broker tests cover policy denial/outage/version mismatch, concurrent lease claims, receipt chaining, wildcard effect matching, all blocking variance classes, operation and observer failures, non-canonical output, expired/over-broad authority, explicit compatibility mode, audit redaction, and rejection of model-reported effects. The independent full release gate passes 127 unit/component/model tests plus all accessibility, workflow, visual, performance, smoke, packaging, build, lint, version, and control gates.

Delivery slice 3 is complete. The portable provider requires a clean primary repository, resolves prospective data paths before creating directories, rejects repository-overlapping links/junctions, creates locked detached worktrees outside the repository, persists validated lifecycle records atomically, inventories bounded create/update/delete/rename effects, and hashes file state without following symbolic links. Promotion requires `ready_to_commit`, at least one action-receipt digest, observable effects, a clean primary workspace, and an unchanged base; it creates a detached candidate and uses fast-forward promotion. Owned worktrees are explicitly unlocked, removed, and checked for disappearance. Interrupted preparing/executing/verifying cells recover as failed for inspection, while stale locks are reported rather than silently deleted.

The provider disables Git hooks and signing for its internal commands, rejects tampered record/candidate metadata, constrains all owned paths, fails on excessive Git/file inspection output, and retains the truthful `securityBoundary = false` classification. It does not claim to isolate hostile processes, filters, or arbitrary commands.

Evidence: 13 focused real-repository tests cover locked preparation, concurrent cell collision, dirty primary rejection, lexical and symlink/junction data-boundary rejection, complete effect inventory, contract/lease/receipt identity, observation snapshots, successful promotion and teardown, stale and dirty promotion rejection, mandatory receipt/effect evidence, interrupted recovery, record tampering, stale locks, and cross-platform-safe identifiers. The independent full release gate passes 140 unit/component/model/integration tests plus every accessibility, workflow, visual, performance, smoke, packaging, build, lint, version, and control gate.

Delivery slice 4 is complete. Run records can optionally carry a bounded execution summary, and run detail renders it through a dedicated operator inspector only when real data exists. The inspector exposes provider and security-boundary claims, finite lifecycle, full base revision, network default, capability scopes, resource ceilings, observed effects, effect variance, verification evidence, and explicit commit/rollback eligibility. Actions remain disabled when the summary or backend callback does not authorize them, with the real blocking reason available in the footer and control title. Legacy runs render exactly as before.

Evidence: three rendered component tests cover semantic structure, truthful worktree warnings, disconnected and connected actions, terminal failure, proof, effects, and variance. Five isolated production workflows verify the populated inspector, guarded promotion, a scoped axe WCAG A/AA scan, and 320 CSS px reflow without horizontal overflow. A full 1440 px populated state was rendered and visually inspected. The independent release gate passes 143 unit/component/model/integration tests, five consequential workflows, six accessibility suites, all 36 existing visual baselines, four performance suites, production smoke, package integrity, build, lint, version, and control validation.

Integration prerequisite complete: compact state polling and paged run history now project run list items without the optional execution summary, while the dedicated run-detail endpoint preserves complete capability, effect, variance, and evidence data. The projection copies rather than mutates stored runs. A seeded production smoke process proves both bounded endpoints omit execution data and the detail endpoint retains it. The full release gate passes 144 tests and all browser, performance, packaging, build, lint, version, and control gates.
