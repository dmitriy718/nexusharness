# Standardize feedback, live state, content, and performance

## User impact

Errors are raw/global, success is often silent, loading looks like failure, and whole-store polling causes broad updates.

## Acceptance criteria

- Errors are categorized and associated with their action or field.
- Loading, offline, reconnecting, success, stale, and empty states are distinct.
- Live updates are scoped or event driven and preserve drafts.
- Long content is safely wrapped, clamped, copied, and expanded by type.

## Dependencies and blockers

Front-end architecture foundation.

## Progress

API failures are categorized as validation, conflict, connection, authorization, missing, or unknown with operator-facing titles, normalized structured messages, retry eligibility, and copyable technical detail. The provider distinguishes booting, online, reconnecting, stale, and offline states, delays short sync indicators to avoid flashing, updates freshness, reacts to browser online/offline events, and uses a slower compact poll. Unchanged store slices retain object identity so unrelated live updates do not reset feature drafts; Settings additionally guards its draft against changed store roots. Success/warning/error/info notices use a status region with separate dismiss controls. Type-specific long content remains clamped in cards, horizontally contained in tables/diffs, line-numbered in previews, and expandable/copyable in run/audit inspectors.

## Verification evidence

- Feedback model perspective: 69 tests across ten files cover structured validation prose, failure categories/retry rules, browser connection errors, freshness labels, and reference-stable store slices. Lint, both TypeScript targets, and production build pass.
- Draft/live perspective: a dirty execution setting stayed at 12 through an unrelated memory/audit mutation and the following compact poll, while freshness returned “Updated now.” This verifies both stable slices and settings-draft protection.
- Offline/recovery perspective: stopping the isolated API produced “Local API offline,” a categorized local-service title, specific proxy message, and Retry/Copy/Dismiss controls without losing the draft. Restart plus Retry restored online/fresh state, removed the error, and preserved the draft. Saving then rendered “Settings saved” in a status toast with a separately named Dismiss notification button.
