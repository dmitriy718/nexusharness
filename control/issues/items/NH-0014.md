# Implement production embedding and vector memory retrieval

## User impact

Memory selection currently recognizes pins and literal task/title words only. Paraphrases and related concepts are missed, ranking is not explainable, and the four-characters-per-token estimate can pack context inaccurately.

## Acceptance criteria

- Real local and remote embeddings are generated, durably stored in a real vector extension, and queried by the Planner runtime path.
- Workspace authorization is enforced inside vector queries and covered by negative tests.
- Create, update, delete, retrospective creation, legacy backfill, model migration, cutover, and rollback have safe indexing semantics.
- Lexical-only, shadow-semantic, hybrid, and diagnostic semantic-only modes are explicit and observable.
- Hybrid scores, diversity, pinned policy, thresholds, and token packing are bounded, deterministic, configurable, and tested.
- Real-model integration, end-to-end prompt assembly, retrieval evaluation, benchmark evidence, migrations, operations, and documentation pass.

## Investigation findings and plan

NexusHarness is a Node/TypeScript/Express/React local-first application using npm and an atomically replaced JSON store. It has no existing database or migration framework, no authentication/tenant model, no telemetry stack, and no unfinished embedding/vector implementation. Memory is globally stored in `store.json`; operator API routes and run retrospectives create it; API/UI routes update, pin, search, and delete it; and `server/agentLoop.ts` synchronously ranks pins plus title/task-type keyword overlap before adding memory only to the Planner prompt. The configured budget uses a four-characters-per-token estimate.

Implementation order:

1. Introduce backward-compatible memory/retrieval configuration, workspace namespaces, preprocessing, typed embedding interfaces/providers, and SQLite migrations.
2. Build a durable generation-separated `sqlite-vec` store, index lifecycle, cache, backfill, re-embedding, cutover, rollback, and health diagnostics.
3. Replace the production selection function with explicit lexical/vector candidate retrieval, normalized hybrid scoring, diversity reranking, pinned policy, and tokenizer-based budget packing.
4. Wire create/update/delete/retrospective lifecycle, diagnostics APIs/UI, CLI operations, evaluation data/runner, benchmark, and complete operations/security documentation.
5. Exercise clean/legacy migrations, real local neural embeddings and vector queries, runtime prompt assembly, all rollout/failure modes, evaluation, performance, full repository gates, and a final false-claim/diff audit.

## Implementation evidence

- Real 384-dimensional MiniLM embeddings are generated locally, persisted in generation-specific sqlite-vec tables, and queried by the Planner runtime; Ollama and OpenAI-compatible HTTP adapters use real provider contracts.
- Workspace namespace filtering is applied before lexical selection and inside vec0 KNN. Cross-workspace tests and the evaluation report zero unauthorized retrievals.
- Create/update/delete/retrospective indexing, atomic batch upsert, resumable backfill, stale invalidation, model cutover, rollback, and guarded pruning are implemented and tested.
- Shadow mode durably records bounded lexical and semantic memory-ID rankings plus overlap-at-K without changing production selection.
- Final release gate: 271 passed, 21 intentional skips; lint/build/accessibility/workflow/visual/performance/smoke/package all passed. npm audit reports zero vulnerabilities.
- Schema 3 migration passed clean creation and legacy fixtures; the live operational round trip 3 -> 2 -> 3 returned sqlite-vec v0.1.9 and quick_check ok.
- Evaluation: hybrid Recall@5 0.9697, MRR 0.8182, nDCG@5 0.8475, unauthorized 0 versus lexical Recall@5 0.4697, MRR 0.4848, nDCG@5 0.4379.
- Benchmark: 1,863.66 batch embeddings/s; retrieval P50/P95 7.61/14.38 ms; vector P50/P95 1.12/1.30 ms; backfill 69.25 memories/s; 2,174,976-byte index for 120 memories; cache hit rate 100% on the measured second pass.
