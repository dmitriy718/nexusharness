# Renovate workspace, memory, and settings workflows

## User impact

The workspace tree implies controls it does not have, memory is difficult to manage, and important settings are unlabeled and easy to lose.

## Acceptance criteria

- Workspace tree is lazy, searchable, expandable, keyboard accessible, and supports preview.
- Memory supports complete creation fields, search, filters, timestamps, and safe deletion.
- Settings are grouped, labeled, validated, and protected by dirty-state handling.

## Dependencies and blockers

Shared components and typed contracts.

## Progress

The workspace workflow is renovated. Root and child directories load independently through realpath-constrained endpoints; server search is breadth-first and bounded to 5,000 visited entries/100 results; file preview reads at most 200 KB and distinguishes binary/truncated content. Symbolic links remain visible but are never followed for preview. The tree supports keyboard expansion and sibling focus, while the detail surface provides breadcrumbs, metadata, line numbers, copy path, reveal parent, and an explicit run-draft context handoff. Memory now supports compound search/kind/task/source filters, pin-first sort choices, complete kind/task/source/title/content/pin editing, created/updated provenance, distinct retrospectives, protected save feedback, and immediate delete with a ten-second restore action. Grouped settings remain the final NH-0006 slice.
