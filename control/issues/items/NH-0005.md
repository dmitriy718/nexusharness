# Redesign runtimes, agents, and MCP tool configuration

## User impact

Forms expose irrelevant fields, rely on placeholders, and give weak progress and capability context.

## Acceptance criteria

- Connector-specific guided forms validate and test before save.
- Agent assignments show runtime and capability compatibility.
- MCP discovery has progress and results.
- Tools expose descriptions, schemas, search, and safe enablement feedback.

## Dependencies and blockers

Shared form system and NH-0007.

## Progress

Runtime and agent configuration are renovated. The connector-specific wizard validates inline, tests through a non-persisting API contract, reports connection stages/latency/discovered models, and requires a separate reviewed save. Configured runtimes show session health, last check, model count, removal impact, and a searchable capability inventory. Agent assignments now identify runtime and native-tool/JSON-fallback compatibility, retain unavailable saved bindings visibly, and use an explicit Save/Discard bar. MCP discovery now advances through server-enforced 500-port chunks with exact progress and cancellation. HTTP/stdio setup exposes only relevant fields and supports guided or raw JSON argument editing. Server detail shows transport, state, capability inventory, refresh time, descriptions, risk/category, input schema, safe enablement, and protected bulk actions. Built-in local tools and approval policy have their own view.

## Verification evidence

- Contract perspective: an isolated mock Ollama returned two real inventory records. `POST /api/runtimes/test` reported latency and models while the runtime count remained zero; the separate save then raised it to one and recorded distinct `runtime.test` and `runtime.add` audit events.
- Model perspective: 47 tests pass across five files, including connector defaults, relevant-field validation, minimal payloads, exact assignment removal impact, and executor capability classification. Lint, both TypeScript builds, the production bundle, and the dependency audit pass.
- Rendered perspective: production Chrome at 1440 px showed one healthy runtime, two models, three health fields, three assignment rows, and the three-stage Configure/Test/Review flow. Switching to llama.cpp CLI removed the endpoint field and exposed only binary/model paths. At 390 px the metrics collapsed to one column, the model table remained internally scrollable, and the body had no horizontal overflow.
- MCP contract perspective: an isolated API accepted a five-port in-range scan, returned exact range/count metadata, rejected an out-of-policy range with HTTP 400, persisted a per-tool enablement update, and recorded distinct discovery/update audit events.
- MCP model perspective: 52 tests across six files cover risk/category precedence, schema summaries, compound search/risk filters, bounded discovery chunks, and validated stdio arguments. The MCP SDK client now advertises the synchronized v2 build identity.
- MCP rendered perspective: production Chrome showed two configuration tabs, two HTTP/stdio servers, descriptions and schema summaries for three selected tools, and five local tools with three approval gates. The stdio form replaced endpoint with command/repeatable arguments and offered raw JSON. A real scan completed at 11/11 ports with results retained. At 390 px metrics and server detail became single-column, tab/page widths remained within the viewport, and body overflow was false.
